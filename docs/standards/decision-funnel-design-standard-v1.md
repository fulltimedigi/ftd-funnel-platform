# FULLTIMEDIGI DECISION FUNNEL DESIGN STANDARD v1

## STATUS

NON-NEGOTIABLE

This standard applies to every Interactive Diagnostic Funnel, Decision Advisor Funnel, Recommendation Engine, Lead Qualification Funnel, Assessment Funnel, and Interactive Lead Magnet built inside the FullTimeDigi ecosystem.

---

# CORE LAW

RESULTS FIRST. QUESTIONS LAST.

Never start by creating questions.

Always start from the final possible outcomes and work backwards.

Correct workflow:

Results
→ Decision Logic
→ Required Signals
→ Questions
→ Recommendation
→ Explanation

Wrong workflow:

Questions
→ Answers
→ Hope a result appears

---

# RULE 1 — RESULT ARCHITECTURE COMES FIRST

Before writing a single question, define all possible final outcomes.

For each result define:

* Result ID
* Result Name
* Result Description
* Who should receive it
* Who should NOT receive it
* Recommended next action
* Priority level

If results are not clearly defined:

STOP.

Do not create questions.

---

# RULE 2 — BUILD ELIGIBILITY BEFORE QUESTIONS

For every result define:

## Eligibility Conditions

What must be true for this result to be recommended?

Examples:

* Experience level
* Qualification level
* Business stage
* Team size
* Goal
* Budget
* Context

---

## Disqualification Conditions

What makes this result inappropriate?

Examples:

* Insufficient experience
* Wrong industry
* Wrong maturity level
* Missing prerequisite

---

Questions must never be written before eligibility and disqualification logic exist.

---

# RULE 3 — SIGNALS BEFORE QUESTIONS

Questions are not the objective.

Signals are the objective.

Define:

What information is required to determine eligibility?

Example:

Need to know:

* Experience
* Qualification
* Goal

These are signals.

Questions are merely a mechanism to collect signals.

---

# RULE 4 — EVERY QUESTION MUST EARN ITS PLACE

For every question ask:

IF this answer changes,

Can the final recommendation change?

OR

Can the ranking between recommendations change?

OR

Can a result be excluded?

OR

Can an offer be optimized?

If the answer is NO:

DELETE THE QUESTION.

No exceptions.

---

# RULE 5 — SEPARATE DECISION SIGNALS FROM OFFER SIGNALS

Decision Signals determine the result.

Examples:

* Experience
* Qualification
* Goal
* Industry
* Current situation

Offer Signals optimize presentation.

Examples:

* Budget
* Live vs Self Study
* Preferred timeline
* Contact preference

Never allow Offer Signals to drive the core recommendation.

Core recommendation must come from Decision Signals.

---

# RULE 6 — BUILD A DECISION MATRIX

Before questions exist, create a Decision Matrix.

For every result define:

* Required signals
* Positive signals
* Negative signals
* Exclusion signals
* Tie-breaker signals

Only after the matrix is complete may questions be created.

---

# RULE 7 — MINIMUM QUESTIONS, MAXIMUM SIGNAL

Do not optimize for more questions.

Optimize for stronger signals.

One high-quality signal is better than three weak questions.

Always ask:

What is the minimum information required to confidently separate the possible outcomes?

---

# RULE 8 — NO FAKE INTELLIGENCE

The engine must never appear smarter than the data collected.

Never generate:

* Personality traits
* Psychological analysis
* Behavioral profiles
* Leadership styles
* Confidence percentages
* Readiness scores
* Strengths or weaknesses

UNLESS those items were explicitly measured.

If the input did not measure it:

The output may not claim it.

---

# RULE 9 — EXPLAIN THE DECISION, NOT THE PERSON

The objective of most business funnels is decision support.

Not personality analysis.

Result structure should prioritize:

1. Recommendation
2. Why it fits
3. Why alternatives were rejected
4. What happens next
5. Recommended action

Focus on explaining the decision.

Do not invent user traits.

---

# RULE 10 — EXPLANATION IS THE VALUE LAYER

A filter provides an answer.

A Decision Funnel provides an answer plus reasoning.

Weak output:

Recommended:
PMP

Strong output:

Recommended:
PMP

Why:

* You meet eligibility requirements
* Your goal aligns with PMP outcomes
* Your experience level supports PMP

Why not CAPM:

* CAPM is designed for earlier-stage professionals

Why not PRINCE2:

* Your context does not prioritize that framework

Next Step:

Enroll in PMP preparation training.

The value comes from clarity, trust, and confidence.

---

# RULE 11 — EVERY RESULT MUST BE DEFENSIBLE

Every statement shown to the user must be traceable to:

* A collected signal
* A decision rule
* A documented eligibility condition

If a statement cannot be defended:

Remove it.

---

# RULE 12 — TIE-BREAKER REQUIRED

If two results are both valid:

The engine must contain an explicit tie-breaker.

Possible tie-breakers:

* Stronger eligibility match
* Better context alignment
* Lower recommendation risk
* Higher goal alignment
* Business-defined priority

Never select results randomly.

---

# TRUST VALIDATION CHECK

Before approving any funnel ask:

1. Are all results clearly defined?

2. Does every question collect a necessary signal?

3. Can every recommendation be defended?

4. Does the result explain the decision?

5. Does the output claim anything not explicitly measured?

If YES:

Remove or rewrite the claim.

FullTimeDigi does not build quizzes.

FullTimeDigi does not build personality tests.

FullTimeDigi builds Interactive Decision Systems.

The goal is not to discover the user.

The goal is to help the user make the right decision with confidence.

RESULTS FIRST.
QUESTIONS LAST.
